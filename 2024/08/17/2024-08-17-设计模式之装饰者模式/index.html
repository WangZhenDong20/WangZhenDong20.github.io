设计模式之装饰者模式

Posted by SleepingFace on Saturday, August 17, 2024

装饰者模式（Decorator Pattern）

概述：

装饰者模式是一种结构型设计模式，它允许你通过将对象放入包含行为的特殊封装对象中来为原对象增加新的行为。这些封装对象被称为装饰者。

关键点：

  • 动态扩展对象功能：装饰者模式在不修改现有类的情况下动态地为对象添加功能。
  • 递归组合：装饰者可以装饰其他装饰者，使得功能扩展非常灵活。
  • 与继承相比的优势：装饰者模式比继承更灵活，因为它允许在运行时动态地添加功能，而继承是在编译时确定的。

具体场景：

举个例子：

有一个点单的需求，有一个点咖啡的场景，咖啡里可以加很多配料，比如牛奶，糖。

这个时候，如果我们增加一种配料巧克力该怎么做呢？

不使用装饰者模式：

1.通过继承和组合的方式

通过继承和组合的方式，为每种咖啡和配料组合创建一个单独的类。这种方法可以工作，但随着配料种类和组合的增加，类的数量会急剧增加，导致类的爆炸性增长。

// 基本咖啡类
class Coffee {
    private String description = "Unknown Coffee";
    private double cost = 0.0;

    public String getDescription() {
        return description;
    }

    public double cost() {
        return cost;
    }

    protected void setDescription(String description) {
        this.description = description;
    }

    protected void setCost(double cost) {
        this.cost = cost;
    }
}

// 具体的咖啡类
class Espresso extends Coffee {
    public Espresso() {
        setDescription("Espresso");
        setCost(1.99);
    }
}

class BlackCoffee extends Coffee {
    public BlackCoffee() {
        setDescription("Black Coffee");
        setCost(1.49);
    }
}

// 具体的配料类
class MilkCoffee extends Coffee {
    public MilkCoffee(Coffee coffee) {
        setDescription(coffee.getDescription() + ", Milk");
        setCost(coffee.cost() + 0.50);
    }
}

class SugarMilkCoffee extends Coffee {
    public SugarMilkCoffee(Coffee coffee) {
        setDescription(coffee.getDescription() + ", Sugar, Milk");
        setCost(coffee.cost() + 0.20 + 0.50);
    }
}

// 使用组合生成具体实例
public class CoffeeShop {
    public static void main(String[] args) {
        Coffee myCoffee = new Espresso();
        myCoffee = new MilkCoffee(myCoffee);
        myCoffee = new SugarMilkCoffee(myCoffee);

        System.out.println("Description: " + myCoffee.getDescription());
        System.out.println("Cost: $" + myCoffee.cost());
    }
}

但是会产生很多问题：

  • 类爆炸：每增加一种新的配料，可能都需要创建多个新类，以处理各种组合。

  • 不灵活：如果想添加新的配料或改变配料的顺序，需要修改或增加大量代码。

  • 难以维护：随着需求的变化，维护和扩展这些类变得非常困难。

2.使用条件语句和标志

class Coffee {
    private String description;
    private double cost;

    private boolean hasMilk;
    private boolean hasSugar;
    private boolean hasChocolate;

    public Coffee(String description, double cost) {
        this.description = description;
        this.cost = cost;
    }

    public void addMilk() {
        hasMilk = true;
        cost += 0.50;
    }

    public void addSugar() {
        hasSugar = true;
        cost += 0.20;
    }

    public void addChocolate() {
        hasChocolate = true;
        cost += 0.70;
    }

    public String getDescription() {
        String fullDescription = description;
        if (hasMilk) fullDescription += ", Milk";
        if (hasSugar) fullDescription += ", Sugar";
        if (hasChocolate) fullDescription += ", Chocolate";
        return fullDescription;
    }

    public double cost() {
        return cost;
    }
}

public class CoffeeShop {
    public static void main(String[] args) {
        Coffee myCoffee = new Coffee("Espresso", 1.99);
        myCoffee.addMilk();
        myCoffee.addSugar();
        myCoffee.addChocolate();

        System.out.println("Description: " + myCoffee.getDescription());
        System.out.println("Cost: $" + myCoffee.cost());
    }
}

这种也会有很多问题：

  • 代码复杂：随着配料的增加，条件语句会越来越复杂，代码的可读性和可维护性会变差。

  • 灵活性不足：如果需要扩展新的功能（如根据用户选择动态增加不同配料），代码会变得更加复杂。

  • 违背开闭原则：每次添加新配料时，都需要修改Coffee类，这违背了“对扩展开放，对修改关闭”的设计原则。

主要解决问题

  • 避免通过继承引入静态特征，特别是在子类数量急剧膨胀的情况下。
  • 允许在运行时动态地添加或修改对象的功能。

使用场景

  • 当需要在不增加大量子类的情况下扩展类的功能。
  • 当需要动态地添加或撤销对象的功能。

实现方式

  • 定义组件接口：创建一个接口，规定可以动态添加职责的对象的标准。
  • 创建具体组件：实现该接口的具体类，提供基本功能。
  • 创建抽象装饰者：实现同样的接口，持有一个组件接口的引用，可以在任何时候动态地添加功能。
  • 创建具体装饰者：扩展抽象装饰者，添加额外的职责。

示例代码

1.创建一个组件接口(Component接口)

// Coffee 接口表示基本的咖啡类型
public interface Coffee {
    String getDescription();
    double cost();
}

2.实现接口的实现类（ConcreteComponent类）

// Espresso 是一种具体的咖啡
public class Espresso implements Coffee {
    @Override
    public String getDescription() {
        return "Espresso";
    }

    @Override
    public double cost() {
        return 1.99;
    }
}

public class BlackCoffee implements Coffee {
    @Override
    public String getDescription() {
        return "Black Coffee";
    }

    @Override
    public double cost() {
        return 1.49;
    }
}

3.创建抽象装饰类（Decorator类）

// Decorator类
public abstract class CoffeeDecorator implements Coffee {
    protected Coffee coffee;

    public CoffeeDecorator(Coffee coffee) {
        this.coffee = coffee;
    }

    @Override
    public String getDescription() {
        return coffee.getDescription();
    }

    @Override
    public double cost() {
        return coffee.cost();
    }
}

4. 创建实体装饰类(ConcreteDecorator类)

// ConcreteDecorator类
public class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public String getDescription() {
        return super.getDescription() + ", Milk";
    }

    @Override
    public double cost() {
        return super.cost() + 0.50;
    }
}

public class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public String getDescription() {
        return super.getDescription() + ", Sugar";
    }

    @Override
    public double cost() {
        return super.cost() + 0.20;
    }
}

public class ChocolateDecorator extends CoffeeDecorator {
    public ChocolateDecorator(Coffee coffee) {
        super(coffee);
    }

    @Override
    public String getDescription() {
        return super.getDescription() + ", Chocolate";
    }

    @Override
    public double cost() {
        return super.cost() + 0.70;
    }
}

5. 使用装饰者模式组合

// 使用装饰者模式组合
public class CoffeeShop {
    public static void main(String[] args) {
        Coffee myCoffee = new Espresso();
        myCoffee = new MilkDecorator(myCoffee);    // 加奶
        myCoffee = new SugarDecorator(myCoffee);   // 加糖
        myCoffee = new ChocolateDecorator(myCoffee);  // 加巧克力

        System.out.println("Description: " + myCoffee.getDescription());
        System.out.println("Cost: $" + myCoffee.cost());
    }
}

这样我想再增加其它配料的时候，就可以只新增ConcreteDecorator类就可以了。

在这个场景中，装饰者模式可以灵活地为咖啡添加不同的配料，而无需为每种可能的组合定义子类。每个配料装饰器都可以动态地叠加在其他装饰器或基本咖啡上，简化了系统的扩展和维护。

总结

在点单中，配料的选择通常是动态的，通过用户输入来决定。在系统中，每种配料是独立的模块，可以在运行时动态组合到咖啡中去。

装饰者模式能够很好地应对配料动态组合的需求，而不需要创建大量类或使用复杂的条件逻辑，保证了系统的灵活性和可扩展性。

通过这种设计，系统能够轻松应对配料种类的增加或变化，并且保持代码的简洁和易维护。

重温一下装饰者模式的核心关键点：

  • 动态扩展对象功能：装饰者模式在不修改现有类的情况下动态地为对象添加功能。
  • 递归组合：装饰者可以装饰其他装饰者，使得功能扩展非常灵活。
  • 与继承相比的优势：装饰者模式比继承更灵活，因为它允许在运行时动态地添加功能，而继承是在编译时确定的。
