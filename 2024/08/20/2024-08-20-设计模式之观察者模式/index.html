设计模式之观察者模式

Posted by SleepingFace on Tuesday, August 20, 2024

观察者模式（Observer Pattern）

概述：

观察者模式（Observer Pattern）是一种行为设计模式，它定义了一种一对多的依赖关系，使得一个对象状态的改变可以通知到所有依赖它的对象，并且自动更新。这种模式特别适用于需要通知多个对象状态变化的场景，常用于实现事件驱动的系统。

关键点：

  • 一个对象状态改变时，自动通知其他依赖对象的问题，同时保持对象间的低耦合和高协作性。

  • 主题（Subject）：也称为被观察者（Observable）。持有观察者列表，提供注册、注销观察者的方法，并在自身状态变化时通知所有观察者。

  • 观察者（Observer）：定义一个接口，包含一个更新方法，当主题状态改变时，主题会调用观察者的更新方法，传递必要的信息。

  • 通知机制：当主题的状态发生变化时，调用通知方法，触发对所有观察者的更新调用。

使用场景：

  • 当一个对象的状态变化需要同时更新其他对象时。

实现方式：

  • Subject（主题）：定义一个观察者管理接口，包括添加、删除观察者的方法，以及通知观察者的方法。

  • ConcreteSubject（具体主题）：实现Subject接口，并在状态变化时通知所有观察者。

  • Observer（观察者）：定义一个更新接口，当主题状态变化时，接收通知并执行相应操作。

  • ConcreteObserver（具体观察者）：实现Observer接口，并在收到通知时执行具体操作。

示例代码：

**场景：**假设我们有一个天气预报系统。这个系统有一个气象站（WeatherStation），它会周期性地更新温度、湿度等数据，并通知注册的显示设备（如手机、电脑、平板等）。这些显示设备作为观察者，需要根据气象站提供的数据来更新自己的显示。

1. 观察者接口Observer

// 观察者接口
interface Observer {
    void update(float temperature, float humidity, float pressure);
}

2. 主题接口Subject

// 主题接口
interface Subject {
    void registerObserver(Observer observer);
    void removeObserver(Observer observer);
    void notifyObservers();
}

3. 具体主题WeatherStation

import java.util.ArrayList;
import java.util.List;

class WeatherStation implements Subject {
    private List<Observer> observers;
    private float temperature;
    private float humidity;
    private float pressure;

    public WeatherStation() {
        observers = new ArrayList<>();
    }

    @Override
    public void registerObserver(Observer observer) {
        observers.add(observer);
    }

    @Override
    public void removeObserver(Observer observer) {
        observers.remove(observer);
    }

    @Override
    public void notifyObservers() {
        for (Observer observer : observers) {
            observer.update(temperature, humidity, pressure);
        }
    }

    // 更新天气数据的方法，并通知所有观察者
    public void setMeasurements(float temperature, float humidity, float pressure) {
        this.temperature = temperature;
        this.humidity = humidity;
        this.pressure = pressure;
        notifyObservers();
    }
}

4. 具体观察者CurrentConditionsDisplay

class CurrentConditionsDisplay implements Observer {
    private float temperature;
    private float humidity;

    @Override
    public void update(float temperature, float humidity, float pressure) {
        this.temperature = temperature;
        this.humidity = humidity;
        display();
    }

    public void display() {
        System.out.println("Current conditions: " + temperature + "°C and " + humidity + "% humidity");
    }
}

5. 客户端代码

public class WeatherStationDemo {
    public static void main(String[] args) {
        // 创建主题（气象站）
        WeatherStation weatherStation = new WeatherStation();

        // 创建观察者（显示设备）
        CurrentConditionsDisplay currentDisplay = new CurrentConditionsDisplay();

        // 注册观察者
        weatherStation.registerObserver(currentDisplay);

        // 模拟气象站的数据变化
        weatherStation.setMeasurements(28.5f, 65.0f, 1013.1f); // Output: Current conditions: 28.5°C and 65.0% humidity
        weatherStation.setMeasurements(27.8f, 70.2f, 1012.5f); // Output: Current conditions: 27.8°C and 70.2% humidity
    }
}

应用场景

  1. 事件驱动系统：如GUI框架中，按钮的点击、文本框的变化等事件监听机制。
  2. 实时数据更新：如股票行情、天气预报等需要实时更新信息的系统。
  3. 订阅-发布系统：如新闻订阅、博客更新通知等，用户可以订阅感兴趣的内容，当内容更新时接收通知。
  4. 日志监听：日志系统中，多个监听器监听日志系统的输出并做相应处理。

观察者模式的变体

  1. 推模型（Push Model）：主题将变化的数据主动推送给观察者，如上面例子中的气象站。
  2. 拉模型（Pull Model）：观察者主动从主题拉取数据。主题只通知有变化的事件，但不提供数据，观察者在接到通知后主动获取所需的数据。

观察者模式的实现细节

  1. 弱引用：在某些语言或框架中，使用弱引用（WeakReference）来防止观察者被意外保留，导致内存泄漏。
  2. 异步通知：在高并发场景下，通知观察者的过程可以异步执行，以避免阻塞。

观察者模式与其他模式的区别

  • 观察者模式 vs. 发布-订阅模式：发布-订阅模式是观察者模式的一种具体实现，通常基于事件通道来广播消息。而观察者模式更注重单一主题的变化通知。
  • 观察者模式 vs. 监听器模式：监听器模式是观察者模式的另一种实现形式，通常在GUI应用中使用，观察者称为监听器，关注特定事件的发生。

总结

观察者模式通过定义一个一对多的依赖关系，使得一个对象状态的变化能够通知多个依赖它的对象。它特别适用于需要动态更新或事件驱动的系统，广泛应用于GUI框架、实时数据更新系统、订阅-发布系统等。尽管观察者模式提供了灵活的解耦机制，但在复杂场景下需要小心处理通知顺序、循环依赖等问题，以避免引入额外的复杂性或性能开销。

重温一下观察者模式的关键点：

  • 一个对象状态改变时，自动通知其他依赖对象的问题，同时保持对象间的低耦合和高协作性。

  • 主题（Subject）：也称为被观察者（Observable）。持有观察者列表，提供注册、注销观察者的方法，并在自身状态变化时通知所有观察者。

  • 观察者（Observer）：定义一个接口，包含一个更新方法，当主题状态改变时，主题会调用观察者的更新方法，传递必要的信息。

  • 通知机制：当主题的状态发生变化时，调用通知方法，触发对所有观察者的更新调用。
