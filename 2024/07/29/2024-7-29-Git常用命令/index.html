Git常用命令

Posted by SleepingFace on Monday, July 29, 2024

Git常用命令

前言： 总是使用idea或者VsCode当中的Git直接操作，但是有时候需要用到git命令的时候往往忘记了就需要现查，于是乎就做了一个小总结。

1.初始化仓库

在一个新的或现有的项目中初始化 Git 仓库：

git init

这会在当前目录中创建一个 .git 目录，Git 将使用该目录来跟踪项目的版本历史。

2.克隆仓库

从远程仓库克隆项目代码到本地：

git clone <repository_url>

此命令将仓库的所有文件和历史记录复制到本地目录。

3.查看当前状态

查看当前工作目录的状态，包括哪些文件已修改、哪些文件在暂存区中：

git status

4.添加文件到暂存区

将指定文件添加到暂存区：

git add <file_name>

也可以添加所有修改过的文件：

git add .

5.提交修改

将暂存区中的更改提交到本地仓库，添加提交消息以说明更改的内容：

git commit -m "Your commit message"

要提交暂存区中和工作区中的所有文件（包括未跟踪的文件），可以使用 -a 参数：

git commit -a -m "Your commit message"

6.查看提交历史

git log

可以使用 --oneline 参数简化输出：

git log --oneline

7. 创建分支

创建一个新的分支，以便在独立的开发环境中工作：

git branch <branch_name>

切换到该分支：

git checkout <branch_name>

你也可以在创建分支的同时切换到该分支：

git checkout -b <branch_name>

8. 合并分支

将一个分支的更改合并到当前分支：

git merge <branch_name>

如果合并过程中出现冲突，Git 会提示你手动解决冲突。

9. 推送到远程仓库

将本地仓库中的更改推送到远程仓库：

git push origin <branch_name>

首次推送分支时，可能需要使用 -u 参数：

git push -u origin <branch_name>

10. 拉取远程更改

从远程仓库拉取最新的更改并合并到当前分支：

git pull

11. 删除分支

删除本地分支：

git branch -d <branch_name>

强制删除未合并的分支：

git branch -D <branch_name>

12. 查看远程仓库

列出所有远程仓库：

git remote -v

13. 添加远程仓库

为本地仓库添加远程仓库：

git remote add origin <repository_url>

14. 移除文件

从工作目录和暂存区移除文件：

git rm <file_name>

如果只想从暂存区移除文件而不删除它：

git rm --cached <file_name>

15. 打标签

为特定的提交打标签，例如发布版本：

git tag <tag_name>

将标签推送到远程仓库：

git push origin <tag_name>

16.撤销&回滚

16.1 git reset

git reset 用于重置当前分支到指定的提交点。它有三种主要模式，根据需要保留或丢弃工作目录的更改。

16.1.1 git reset --soft

此模式会将分支指针移动到指定的提交，但保留暂存区和工作目录中的更改。

git reset --soft <commit版本号>

使用场景：回滚提交但保留文件更改，适用于想要重新提交的情况。

16.1.2 git reset --mixed

这是 git reset 的默认模式。它会将分支指针移动到指定提交，同时保留工作目录中的更改，但会清空暂存区。

git reset --mixed <commit版本号>

使用场景：回滚提交并清除暂存区，但保留文件更改，适用于需要修改文件后再提交的情况。

16.1.3 git reset --hard

此模式将分支指针、暂存区和工作目录都重置到指定的提交状态。所有未提交的更改将被丢弃。

git reset --hard <commit版本号>

使用场景：彻底回滚到某个提交，丢弃所有更改。注意：此操作不可逆，请谨慎使用。

16.2 git revert

git revert 用于撤销指定的提交，生成一个新的提交来记录撤销操作。与 git reset 不同，git revert 不会修改提交历史。

16.2.1 撤销单个提交

git revert <commit版本号>

此命令会创建一个新的提交，该提交的更改与指定的提交相反，从而撤销其影响。

使用场景：撤销特定的提交但保留历史记录，适用于协作开发中需要保留提交历史的情况。

16.2.2 撤销多个提交

如果需要撤销多个连续的提交，可以使用 git revert 的范围功能：

git revert <commit1>..<commit2>

此命令会逐个创建新提交来撤销从 <commit1> 到 <commit2>（不含）的所有提交。

16.2.3 使用 --no-commit 选项

使用 --no-commit 选项，git revert 将不会自动创建新的提交，而是将更改保留在暂存区，以便你可以在进一步修改后再提交。

git revert --no-commit <commit>

17.合并分支 —— git merge & git rebase

17.1 git merge

git merge 是一种将两个分支的历史记录合并在一起的操作。它创建一个新的“合并提交”（merge commit），将两个分支的代码整合到一起。当你在一个分支 v1 上执行 git merge master 时，Git 会创建一个新的合并提交，包含 v1 分支和 master 分支的所有历史记录。

git checkout v1
git merge master

优点：

  • 保持完整的历史记录：git merge 保留了所有分支的历史记录，包括每个分支的独立开发历史。
  • git merge 不会改变现有的提交历史，非常适合团队协作。

缺点：

  • 额外的合并提交：每次合并操作都会生成一个合并提交，尤其是在频繁合并的情况下可能会使提交历史变得复杂。
  • 由于保留了两个分支的历史，查看提交历史时可能会显得混乱。

示例：

假设有如下分支情况：

A---B---C v1
     \
      D---E master

执行 git merge master 后，历史会变成：

A---B---C---F v1 (merge commit)
     \     /
      D---E master

使用场景：

  • 需要保留分支的完整历史记录。

  • 在团队协作时，避免修改公共分支的历史。

17.2 git Rebase

git rebase 是一种将分支的更改“重新播放”到另一个分支之上的操作。它通过改变提交的基础来实现将分支整合到一起，从而生成线性历史。当你在分支 v1 上执行 git rebase master 时，Git 会将 v1 分支上的提交 A, B, C 依次应用到 master 分支的最新提交之后，形成一个新的线性历史。

git checkout v1
git rebase master

优点：

  • 线性历史：git rebase 生成的历史记录是线性的，没有合并提交，提交历史看起来更加干净整洁。

  • 便于查看历史：由于历史是线性的，使用 git log 等命令查看历史时更加直观。

缺点：

  • 修改历史：git rebase 会重写提交历史，这可能会导致提交哈希值改变，因此不适合在已共享的分支上使用。
  • 复杂的冲突处理：如果存在冲突，每个提交都可能会引发冲突，需要逐一解决。

示例：

假设有如下分支情况：

A---B---C v1
     \
      D---E master

执行 git rebase master 后，历史会变成：

D---E---A'---B'---C' v1

其中 A', B', C' 是重新基于 master 的新提交。

使用场景：

  • 清理提交历史，让历史记录更为简洁。

  • 在自己的分支上重写历史，但不推荐在已经共享的分支上使用。
