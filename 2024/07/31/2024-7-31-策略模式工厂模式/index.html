策略模式+工厂模式实现多种文件类型上传处理

Posted by SleepingFace on Wednesday, July 31, 2024

策略模式+工厂模式实现多种文件类型上传处理

前言： 记录在SleepingFace BI项目里的一个小优化，采用策略模式+工厂模式代替了if-else实现多种文件类型的上传处理。

1. 策略模式

1.1 简介

策略模式定义了一系列算法，并将每个算法封装起来，使得它们可以互换。策略模式使得算法的变化独立于使用它们的客户端。

1.2 应用场景

  • 需要在运行时根据不同条件选择不同的算法。

  • 当一个类需要使用多个算法或行为，且不希望客户端直接干预时。

  • 需要消除条件语句（if-else 或 switch-case）来选择行为时。

1.3 实现方法

接口或抽象类：定义一系列算法的接口或抽象类。

具体策略类：实现不同的算法。

环境类：维护对策略的引用，用于在运行时决定使用哪个策略。

1.4 优缺点

  • 可以自由切换算法，灵活性高。

  • 可以避免使用大量的条件语句。

  • 将算法的实现细节与使用算法的代码分离，遵循了开闭原则（对扩展开放，对修改封闭）。

  • 但是可能会产生大量的策略类，增加系统的复杂性。

2. 工厂模式

2.1 简介

工厂模式提供一个创建对象的接口，而不是让客户端直接实例化对象。通过工厂模式，客户端可以通过接口或抽象类来创建具体的对象实例。

2.2 应用场景

  • 需要通过子类来决定创建的对象类型时。

  • 对象的创建过程复杂或需要管理时。

  • 想要对实例化过程进行控制时。

2.3 实现方法

简单工厂模式：提供一个工厂类，根据传入的参数决定创建的对象类型。

工厂方法模式：定义一个接口或抽象类，由子类决定实例化哪个类。

抽象工厂模式：提供一个接口，用于创建一系列相关或依赖的对象，而无需指定具体类。

2.3.1 简单工厂模式

简单工厂模式又称为静态工厂方法模式，使用一个工厂类根据输入的参数决定创建哪一个具体的类的实例。

  • 抽象产品：定义了产品的规范，描述了产品的主要特性和功能。
  • 具体产品：实现或者继承抽象产品的子类。
  • 具体工厂：提供了创建产品的方法，调用者通过该方法来获取产品。

class Product {
    void display() {}
}

class ConcreteProductA extends Product {
    @Override
    void display() {
        System.out.println("Product A");
    }
}

class ConcreteProductB extends Product {
    @Override
    void display() {
        System.out.println("Product B");
    }
}

class SimpleFactory {
    static Product createProduct(String type) {
        if (type.equals("A")) {
            return new ConcreteProductA();
        } else if (type.equals("B")) {
            return new ConcreteProductB();
        }
        return null;
    }
}

优点：

  • 客户端只需知道传递的参数即可，不需要关心具体的类名和实例化过程。

  • 工厂类集中管理对象的创建，方便维护和修改。

缺点：

  • 违反开闭原则（Open/Closed Principle），因为每增加一个新产品，都需要修改工厂类。
  • 当产品种类过多时，工厂类可能会变得非常庞大和复杂。

2.3.2 工厂方法模式

工厂方法模式定义了一个创建对象的接口，但由子类决定要实例化的类是哪一个。工厂方法模式将对象的创建过程延迟到子类中。

  • 抽象工厂类：声明工厂方法，用于创建产品对象。
  • 具体工厂类：实现工厂方法，返回具体的产品实例。
  • 产品接口：工厂方法所创建对象的接口。
  • 具体产品类：实现产品接口的具体类。

interface Product {
    void display();
}

class ConcreteProductA implements Product {
    @Override
    public void display() {
        System.out.println("Product A");
    }
}

class ConcreteProductB implements Product {
    @Override
    public void display() {
        System.out.println("Product B");
    }
}

abstract class Factory {
    abstract Product createProduct();
}

class ConcreteFactoryA extends Factory {
    @Override
    Product createProduct() {
        return new ConcreteProductA();
    }
}

class ConcreteFactoryB extends Factory {
    @Override
    Product createProduct() {
        return new ConcreteProductB();
    }
}

优点：

  • 遵循开闭原则，可以通过增加新的工厂类来扩展产品的创建，而无需修改现有代码。

  • 更加灵活，可以通过不同的工厂类创建不同的产品对象。

缺点：

  • 需要为每种产品创建一个对应的工厂类，增加了系统的复杂性。

2.3.3 抽象工厂模式

抽象工厂模式提供一个接口，用于创建一系列相关或互相依赖的对象，而无需指定具体的类。它允许客户端使用抽象接口来创建一组相关的产品。

抽象工厂接口：声明创建产品的方法，每个方法负责创建不同种类的产品。

具体工厂类：实现抽象工厂接口，负责创建具体的产品对象。

产品接口：每种产品类型的接口。

具体产品类：实现产品接口的具体类。

interface ProductA {
    void display();
}

interface ProductB {
    void display();
}

class ConcreteProductA1 implements ProductA {
    @Override
    public void display() {
        System.out.println("Product A1");
    }
}

class ConcreteProductB1 implements ProductB {
    @Override
    public void display() {
        System.out.println("Product B1");
    }
}

class ConcreteProductA2 implements ProductA {
    @Override
    public void display() {
        System.out.println("Product A2");
    }
}

class ConcreteProductB2 implements ProductB {
    @Override
    public void display() {
        System.out.println("Product B2");
    }
}

interface AbstractFactory {
    ProductA createProductA();
    ProductB createProductB();
}

class ConcreteFactory1 implements AbstractFactory {
    @Override
    public ProductA createProductA() {
        return new ConcreteProductA1();
    }
    @Override
    public ProductB createProductB() {
        return new ConcreteProductB1();
    }
}

class ConcreteFactory2 implements AbstractFactory {
    @Override
    public ProductA createProductA() {
        return new ConcreteProductA2();
    }
    @Override
    public ProductB createProductB() {
        return new ConcreteProductB2();
    }
}

优点：

  • 可以创建一组相关的产品，保证产品之间的一致性。

  • 遵循开闭原则，可以通过增加新的具体工厂来扩展产品族。

缺点：

  • 增加了系统的抽象性和复杂性，因为需要定义多个产品接口和工厂接口。

  • 当需要扩展一个新的产品时，所有的工厂类都需要修改。

案例：利用策略模式+工厂模式实现多种文件类型上传处理

1. 创建抽象策略类

/**
 * 抽象策略类
 */

public interface FileReaderStrategy {
    ArrayList<String> readFile(MultipartFile file) throws IOException;
}

2.定义策略类

我这里实现了doc、docx、markdown、txt的读取策略，把他们交给Spring容器管理。

@Component
public class DocFileReaderStrategy implements FileReaderStrategy {
    @Override
    public ArrayList<String> readFile(MultipartFile file) throws IOException {
        return TxtUtils.readerDocFile(file);
    }
}

@Component
public class DocxFileReaderStrategy implements FileReaderStrategy {
    @Override
    public ArrayList<String> readFile(MultipartFile file) throws IOException {
        return TxtUtils.readerDocxFile(file);
    }
}

@Component
public class MdFileReaderStrategy implements FileReaderStrategy {
    @Override
    public ArrayList<String> readFile(MultipartFile file) throws IOException {
        return TxtUtils.readerFile(file);
    }
}

@Component
public class TxtFileReaderStrategy implements FileReaderStrategy {
    @Override
    public ArrayList<String> readFile(MultipartFile file) throws IOException {
        return TxtUtils.readerFile(file);
    }
}

3.创建策略工厂

3.1 通过读取yml配置文件来获取有哪些策略

@Getter
@Setter
@Configuration
@ConfigurationProperties(prefix = "readfile")
public class ReadFileTypeConfig {

    private Map<String,String> types;

}

3.2 在yml里进行配置

readfile:
  types:
    doc: docFileReaderStrategy
    docx: docxFileReaderStrategy
    md: mdFileReaderStrategy
    txt: txtFileReaderStrategy

3. 策略工厂

/**
 * 操作策略的上下文环境类 工具类
 * 将策略整合起来 方便管理
 */
@Component
public class FileReaderFactory implements ApplicationContextAware {

    @Resource
    private ReadFileTypeConfig readFileTypeConfig;

    private static Map<String, FileReaderStrategy> strategyMap = new ConcurrentHashMap<>();


    /**
     * 从配置文件中读取策略信息存储到map中
     * {
     *     doc: docFileReaderStrategy
     *     docx: docxFileReaderStrategy
     *     md: mdFileReaderStrategy
     *     txt: txtFileReaderStrategy
     * }
     *
     * @param applicationContext
     * @throws BeansException
     */
    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        readFileTypeConfig.getTypes().forEach((k, y) -> {
            strategyMap.put(k, (FileReaderStrategy) applicationContext.getBean(y));
        });
    }

    /**
     * 对外提供获取具体策略
     */
    public FileReaderStrategy getStrategy(String type) {
        FileReaderStrategy fileReaderStrategy = strategyMap.get(type);
        return fileReaderStrategy;
    }
}

其中setApplicationContext函数从Spring容器中获取策略的Bean对象到Map当中。

getStrategy函数根据对应类型获取对应的策略。

4. 改造service里的代码

我们注入FileReaderFactory，之后调用fileReaderFactory.getStrategy(suffix)获取策略。

判断策略是否存在，如果存在就可以调用具体策略的方法strategy.readFile；否则可以抛出异常。

@Component
public class FileProcessor {

    @Resource
    private FileReaderFactory fileReaderFactory;

    public ArrayList<String> processFile(String suffix, MultipartFile multipartFile) throws IOException {

        FileReaderStrategy strategy = fileReaderFactory.getStrategy(suffix);

        if (strategy != null) {
            return strategy.readFile(multipartFile);
        } else {
            throw new IllegalArgumentException("Unsupported file type: " + suffix);
        }
    }
}
